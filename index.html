How is a performance health rating calculated?

A database with healthy performance spends it time productively - on CPU and I/O.

A database with unhealthy performance often spends too much time waiting for bottlenecks such as locks and latches, which throttle the databases throughput of data.

By analysing the waits of an SQL Server database we can determine the relative proportion of time it spends on CPU, I/O, Locks, Latches, Redo, Data Guard etc. As long as a database devotes enough time to CPU and I/O, it will be considered healthy.

Additionally, the single-block-read latency of the database is taken into consideration. This needs to be low/fast enough to indicate that the time being spent on I/O is productive.

However, if the “DB Time” of the database is very low, there is not enough load being placed on the database to assess its health.

So in a nutshell, if a database devotes most of its time to CPU and I/O and has a low single-block read latency, it will obtain a performance-health rating of 80% or more, which is considered healthy.

How much history is considered for a rating?

Spotlight on SQL Server samples performance metrics of your database every six hours.

Also, the Spotlight Developer samples metrics every hour.

The performance-health rating of the database is the current (most recent) value of this moving average.

A rating of 80% or more is considered healthy.

These hourly health-ratings and the moving average are shown in a chart on this page.

About Samples

Spotlight on SQL Server samples the performance metrics of the database every six hours. Each sample is then assigned a performance-health rating. A rating of 80% or more is healthy and appears as a green dot in this chart. If a sample is assigned an unhealthy rating, it is appears as a red dot.

About the Moving Average

If the individual samples fluctuate a lot, the information in the chart would be difficult to interpret - the chart would just be a noisy collection of dots. A moving average is generated to smooth out fluctuations in the individual samples, producing a line with that can be interpreted more easily.

Note: The moving average line will not appear in the chart until there are more than 40 individual samples.

The performance of each SQL Server instance is designated as either healthy or not, based on the proportion of time spent waiting for CPU+I/O versus other wait-types, in conjunction with the latency of single-page-reads. An instance that spends 80% of its time waiting for CPU or I/O and has low I/O latency is designated as healthy.

These attributes indicate good performance health, as they imply that the instance is not bottlenecked in a way that prevents it from being able to perform work and be responsive.

CPU and I/O

Spending time on CPU and I/O is the best indicator that the database is performing useful work. In the chart these bars are displayed in green, to signify that having the database spend time here is good. For example, an OLAP database may spend most time on CPU while an OLTP database may spend most time on I/O. Either way, you ideally want at least 95% of all time-spent-waiting to be on CPU and I/O.

Other overheads

The red bars indicate the proportion of time spent by the database on overheads. The red color signifies that having the database spend time here is bad. A healthy database will not spent more than 5% of its time on overheads, with no single overhead exceeding 2%.

I/O Latency

Latency of single-block reads is a good indicator of the responsiveness of the I/O subsystem.

The latency should be less than or equal to 10ms in a healthy database, so that it can perform enough I/O’s per second to feel responsive.
